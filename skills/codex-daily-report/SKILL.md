# Codex Daily Report

## Purpose

Produce a concise Chinese daily report from local Codex session history. The default final output is a short submitted-style summary, while raw and long reports are retained only as history/evidence.

## Default End-to-End Flow

When the user asks to generate or test today's daily report, do this by default:

1. Check the report schedule with `scripts/china_workday.py --date YYYY-MM-DD --schedule`.
2. Generate a filtered raw evidence report with `scripts/codex_session_daily_report.py`.
3. Read `references/prompt.md` and write a 3-4 line short submitted report using `references/submission-summary-format.md`.
4. Save the short daily draft as `codex-daily-submit-YYYY-MM-DD.md`.
5. Append/update the day entry in `codex-daily-submit-YYYY-MM.md` with `scripts/monthly_submission_report.py`.
6. If the user asks to send it, send the short daily draft with `scripts/send_wecom_report.py`.
7. Return only the short summary and saved paths unless the user asks for long/raw detail.

## Workflow

1. Determine the report date and timezone.
   - Use the user-provided date when present.
   - Otherwise use today's local date.
   - Default timezone to `Asia/Shanghai` unless the user says otherwise.

2. Check China workday status before doing the report work.
   - Default policy file: `references/china-workday-policy.json`.
   - Run `scripts/china_workday.py --date YYYY-MM-DD --schedule`.
   - For a daily report, continue only when `schedule.run_daily_report` is true.
   - For a weekly report, continue when `schedule.run_weekly_report` is true; this defaults to Sunday, the last day of the weekend.
   - For a monthly report, continue when `schedule.run_monthly_report` is true; this is the last China workday of the month.
   - If the user explicitly says the rest day should be included, such as "今天我也上班" or "人工指定加入", pass `--include-rest-day` to both `china_workday.py` and `monthly_submission_report.py`.
   - Keep company-specific exceptions in `references/china-workday-policy.json`; `work_dates` override weekends, and `rest_dates` override normal weekdays.
   - For future years, update the policy from the official holiday notice before relying on automatic gating.

3. Locate the raw session report generator.
   - Default to this skill's bundled script: `scripts/codex_session_daily_report.py`.
   - Use an external script only when the user explicitly provides one or asks to compare/update it.
   - If the bundled script is missing, tell the user the skill is incomplete before searching for a fallback.

4. Maintain the noise-session exclusion list.
   - Default exclusion file: `references/session-exclusions.json`.
   - Use exact `thread_ids` for one-off noisy sessions.
   - Use `title_patterns`, `cwd_patterns`, or `user_message_patterns` for recurring noise such as report self-tests, blank sessions, or skill maintenance.
   - Update the list when the user explicitly says a session is noise.
   - It is also acceptable to update the list without asking when the session is clearly generated by this skill's own testing or is an empty no-content session.
   - Do not exclude real work just because it is small, interrupted, or inconclusive; put it in "待跟进" instead.

5. Choose output paths.
   - Prefer an explicitly provided `--output` path.
   - If an automation or runtime config provides an output root, create/use the report month directory under it: `{output_root}/YYYY-MM/`.
   - Put all persistent files for a report date in that month directory.
   - Otherwise write the raw evidence report to the current working directory as `codex-session-report-{YYYY-MM-DD}.md`.
   - If the user provides a local archive workspace, writing under its `{YYYY-MM-DD}/api/` directory is acceptable.
   - Short daily draft defaults to `codex-daily-submit-{YYYY-MM-DD}.md`.
   - Monthly submitted report defaults to `codex-daily-submit-{YYYY-MM}.md`.
   - Create the parent directory if needed.

6. Run the generator.

```bash
python3 /path/to/skill/scripts/codex_session_daily_report.py --date YYYY-MM-DD --tz Asia/Shanghai --output /path/to/codex-session-report-YYYY-MM-DD.md
```

The bundled script is self-contained and uses only the Python standard library. It scans `~/.codex/sessions` and, by default, `~/.codex/archived_sessions`; pass `--codex-home` when Codex data lives elsewhere. It automatically applies `references/session-exclusions.json`; pass a runtime `--exclude-file /path/to/session-exclusions.json` when the automation provides an external exclusion file, pass `--exclude-file ""` to disable filtering, or `--show-excluded` to audit hidden sessions.

7. Read the generated Markdown and summarize it.
   - First read `references/prompt.md` and apply its evidence, certainty, and tone rules.
   - Preserve factual counts: active sessions, user task changes, total turns, unfinished turns, shell events, web events.
   - Identify workstreams by session title, user requests, and result summaries.
   - Separate completed work from in-progress or blocked work.
   - Group sessions into "what the day was mainly spent on"; do not produce a session-by-session diary unless the user asks for raw details.
   - Merge related sessions across DBC, SQL, webhook, deployment, and reporting topics into a small set of major workstreams.
   - Include file paths, URLs, services, thread ids, or commands only when useful for follow-up.
   - Do not invent outcomes beyond the raw report.

8. Deliver according to the user's intent.
   - Default final answer should be the short submitted-style summary, 3-4 lines.
   - Save long/internal reports only for history and evidence; do not print them by default.
   - If they asked to "生成日报" or "写日报", provide the short summary and mention where long/raw history was saved.
   - If they asked for "提交版", "日报提交", or "我需要提交的版本", use `references/submission-summary-format.md` and produce a short business-facing worklog.
   - If they ask to save or maintain submitted reports, append/update the submitted worklog in the month file with `scripts/monthly_submission_report.py`.
   - If they ask for weekly records or weekly summary, summarize from daily submitted reports with `references/weekly-summary-format.md`, then append/update the ISO week entry in the month weekly file with `scripts/weekly_submission_report.py`.
   - If they ask for monthly records or monthly summary, summarize from the month daily submitted report with `references/monthly-summary-format.md`; default draft path is `codex-monthly-submit-YYYY-MM.md`.
   - If they ask to send the daily report, send only the short submitted report with `scripts/send_wecom_report.py`; use `references/wecom-notification.md`.
   - If they asked only to run the script, report the generated file and key counts.
   - If they asked to create/update an automation, use the app automation tool instead of writing manual scheduler instructions.

## Monthly Submitted Reports

Use one Markdown file per month for the final short submitted reports. The month file is named `codex-daily-submit-YYYY-MM.md` by default.

Append or update a day entry:

```bash
python3 /path/to/skill/scripts/monthly_submission_report.py --date YYYY-MM-DD --content-file /path/to/daily-submit.md --output-dir /path/to/monthly-reports
```

Rules:

- A date section is marked as `## YYYY-MM-DD（周X，工作日）`, `调休工作日`, or `休息日-人工加入`.
- If the date is a rest day and the user explicitly asks to include it, pass `--include-rest-day`; the section is marked `休息日-人工加入`.
- By default, China workday status is read from `references/china-workday-policy.json`; adjusted weekend workdays are included and official holidays are skipped.
- Maintain adjusted workdays, holidays, and company-specific exceptions in `references/china-workday-policy.json` through `work_dates` and `rest_dates`.
- Daily submitted month files are root report data and are retained for 12 months by default; older `codex-daily-submit-YYYY-MM.md` files in the same output directory are removed by the script.

## Weekly Submitted Reports

Use one Markdown file per month for weekly submitted reports. The file is named `codex-weekly-submit-YYYY-MM.md` by default. Do not create yearly weekly summary files.

Create a short weekly draft from the daily submitted reports, then append/update the week entry:

```bash
python3 /path/to/skill/scripts/weekly_submission_report.py --week-date YYYY-MM-DD --content-file /path/to/codex-weekly-submit-YYYY-Www.md --output-dir /path/to/weekly-reports
```

Rules:

- A week section is marked as `## YYYY-Www（YYYY-MM-DD 至 YYYY-MM-DD）`.
- Re-running the same ISO week updates that week section instead of appending duplicates.
- Weekly summaries should be derived from short daily submitted reports whenever possible, not raw session details.
- Weekly summary uses `第一项：标题` paragraph style by default.
- Next-week plan uses 3 item-paragraphs by default.
- Weekly month files are retained for 12 months by default.

## Monthly Summary Reports

Generate a monthly summary on the last China workday of the month, using the current month's daily submitted report as source evidence.

Rules:

- Default source: `codex-daily-submit-YYYY-MM.md`.
- Default monthly summary draft: `codex-monthly-submit-YYYY-MM.md`.
- Read `references/monthly-summary-format.md` before writing the monthly summary.
- Monthly summaries should be derived from short daily submitted reports whenever possible, not raw session details.
- Do not overwrite the daily month file; monthly summaries are separate files.
- Monthly summaries are retained for 12 months by policy.

## Automation Schedule

For the `daily-report` automation, trigger once per day and let the skill decide what to generate:

- Daily report: generate only when `scripts/china_workday.py --schedule` returns `schedule.run_daily_report=true`.
- Weekly report: generate on Sunday when `schedule.run_weekly_report=true`.
- Monthly report: generate on the last China workday of the month when `schedule.run_monthly_report=true`.
- Reports are dependency ordered, not conflicting: generate and append the daily report first when it is due, then generate weekly and monthly summaries from the updated monthly daily report.
- Multiple outputs may be generated on the same run, for example daily + monthly on the last workday, or weekly + monthly if month end falls on Sunday.

## Runtime Config Integration

Some deployments provide runtime configuration outside the packaged skill, such as an output root directory, a custom session exclusion file, or a webhook URL. Treat those runtime values as higher priority than bundled defaults, but do not copy local paths, webhook keys, or user-specific IDs into packaged skill files.

Read `references/initial-setup.md` when configuring a new automation or machine.

When an output root is provided:

- Compute `YYYY-MM` from the report date.
- Create `{output_root}/YYYY-MM/`.
- Save raw evidence, single-day daily drafts, monthly daily root data, weekly records, and monthly summaries for that report date under `{output_root}/YYYY-MM/`.
- For cross-month weekly reports, read only the daily root files from the two involved month directories.

When an external exclusion file is provided:

- Pass it to `scripts/codex_session_daily_report.py` with `--exclude-file`.
- Keep `references/session-exclusions.json` as the portable bundled fallback.

## File Layers

Maintain these report layers when saving files:

- Final short summary/root data: `codex-daily-submit-YYYY-MM.md` monthly file, with each day as a date section. This is the user's primary output and should be retained for 12 months.
- Optional single-day short draft: `codex-daily-submit-YYYY-MM-DD.md`; use only as an intermediate before appending/updating the month file.
- Weekly record: `codex-weekly-submit-YYYY-MM.md` monthly file, with each ISO week as a date-range section.
- Monthly summary: `codex-monthly-submit-YYYY-MM.md`; derived from the month daily submitted report.
- Do not create yearly report files; monthly summaries are the highest-level retained report.
- History/evidence: raw or long reports such as `codex-session-report-YYYY-MM-DD.md`; keep them for traceability, but do not present them as the final result unless requested.

## Context Loading

- Treat `codex-daily-submit-YYYY-MM.md` as root report data, not default conversation context.
- Do not load historical daily month files into context unless the user explicitly asks to query, compare, regenerate, or summarize history.
- Normal daily generation should use today's raw evidence and the current month file only for append/update mechanics.
- Weekly generation should load only the month files needed for that ISO week, including the adjacent month when the week crosses a month boundary.
- Monthly generation should load only the target month's daily submitted file.

## WeCom Sending

Send only the final short report to WeCom. Do not send raw session evidence or long/internal summaries.

```bash
python3 /path/to/skill/scripts/send_wecom_report.py \
  --date YYYY-MM-DD \
  --content-file /path/to/codex-daily-submit-YYYY-MM-DD.md \
  --webhook-url "$WECOM_WEBHOOK_URL"
```

The sending script also checks China workday status when `--date` is provided. Use `--dry-run` when validating formatting. The webhook URL may be provided by the user or via `WECOM_WEBHOOK_URL`; avoid hardcoding webhook keys in generated skill files.

## Summary Style

Always apply `references/prompt.md` before writing any polished or submitted report. Default to `references/submission-summary-format.md` for the daily final user-facing output. Use `references/weekly-summary-format.md` for weekly records and `references/monthly-summary-format.md` for monthly records. Use `references/daily-summary-format.md` only for long/internal historical summaries with overview, outputs, risks, and follow-ups.

## Bundled Resources

- `scripts/codex_session_daily_report.py`: generate the raw Markdown session report from local Codex JSONL session files and the optional `state_5.sqlite` thread-title database.
- `scripts/china_workday.py`: determine whether a date is a China workday and output automation schedule flags for daily, weekly, and monthly reports.
- `scripts/monthly_submission_report.py`: append/update submitted daily report entries in one Markdown file per month and prune old month files.
- `scripts/weekly_submission_report.py`: append/update submitted weekly report entries in one Markdown file per month and prune old month files.
- `scripts/send_wecom_report.py`: send the final short Markdown report to a WeCom bot webhook, with optional date-based China workday gating.
- `references/session-exclusions.json`: maintain noise-session exclusions used by the bundled script.
- `references/china-workday-policy.json`: configure official China holidays, adjusted workdays, company-specific date exceptions, month-file prefix, and retention months.
- `references/initial-setup.md`: runtime setup guide for output root, external exclusion file, WeCom webhook, directory layout, and retention policy.
- `references/prompt.md`: quality prompt for evidence handling, certainty control, research tone, and submitted report style.
- `references/wecom-notification.md`: sending rules and command pattern for WeCom bot delivery.
- `references/daily-summary-format.md`: structure for the polished Chinese daily summary.
- `references/submission-summary-format.md`: structure and tone rules for the short submitted daily worklog.
- `references/weekly-summary-format.md`: structure and tone rules for weekly worklog records.
- `references/monthly-summary-format.md`: structure and tone rules for monthly worklog records.

## Quality Checks

- Confirm the China workday check passed before generating/appending/sending a daily report, unless the user explicitly requested `--include-rest-day`.
- Confirm the generated Markdown exists and is non-empty.
- Apply `references/prompt.md`; reduce deterministic assertions and use research tone for investigation work.
- Final user-facing report should be 3-4 lines unless the user asks for detail.
- Database/slow SQL/alert items should preserve enough evidence to locate the issue: instance/server, database, table/object, SQL fragment or SQL_ID, and status when available.
- Do not send or display raw session evidence as the final report.
- If the raw report has `未完成/进行中回合`, reflect them in "待跟进".
- If a session title is vague, infer cautiously from the user message and keep the thread id for traceability.
- If the generator fails, inspect the error and the script arguments before changing code.
